# Introduction

**Feature Gate** is a collection of SDKs for TypeScript, Rust, Python, Go, and other languages. It works as a unified abstraction layer for evaluating feature flags and reading remote configurations. The goal is to give your application a single, stable API that hides the complexity of whatever infrastructure provider you are using underneath.

## Two-layer architecture

The system is built around a clear separation of responsibilities with two main components:

1. **Provider layer** — A swappable adapter that connects to the underlying service or rules engine. This can be a third-party platform like Statsig or Unleash, an internal solution, or an in-memory implementation for testing.
2. **Management Service layer** — The stable interface that your application's business logic talks to. It is initialized from a provider, which fully isolates your application code from external dependencies.

## Key capabilities

### Safe fallbacks

An application should never break because of a failure in the feature flag system. Any network error, provider outage, or internal exception is caught silently by the library. When a failure occurs, or when the system has not been initialized yet, the library automatically returns safe default values — for example, reporting a feature as "off" or returning empty configurations.

### Contextual evaluation

You can attach detailed context to each evaluation request. This context can include user identifiers, session identifiers, IP addresses, runtime environments, and custom properties. This allows the underlying provider to apply precise segmentation rules and deliver personalized experiences to different groups of users.

### Synchronous and asynchronous modes

Every query operation is available in both modes:

- **Synchronous** — Resolves evaluations immediately based on the local state or cache from the last provider update. Ideal for non-blocking use.
- **Asynchronous** — Forces a fetch of the most recent data directly from the source, accepting the inherent network latency.

### Explicit lifecycle management

The system gives you clear control over startup and shutdown. A provider must be explicitly initialized — where authentication and the initial state download happen — before the main service is instantiated. Shutdown methods are also exposed to ensure clean closure of connections and background processes.

### Extensibility and testing support

Because the design is provider-agnostic, you can switch feature flag providers by changing only the adapter at application startup, with no changes to your business logic. The library also includes native support for local development and automated testing through in-memory mock providers.
